    <div id="mw-content-text" lang="en" dir="ltr" class="mw-content-ltr">
     <p><b>Address Resolution Protocol</b> (<b>ARP</b>) is a <a href="/wiki/Telecommunication" title="Telecommunication">telecommunications</a> protocol used for resolution of <a href="/wiki/Network_layer" title="Network layer">network layer</a> addresses into <a href="/wiki/Link_layer" title="Link layer">link layer</a> addresses, a critical function in multiple-access networks. ARP was defined by <a class="external mw-magiclink-rfc" href="//tools.ietf.org/html/rfc826">RFC 826</a> in 1982.<sup id="cite_ref-1" class="reference"><a href="#cite_note-1"><span>[</span>1<span>]</span></a></sup> It is <a href="/wiki/Internet_Standard" title="Internet Standard">Internet Standard</a> STD 37. It is also the name of the program for manipulating these addresses in most operating systems.</p> 
     <p>ARP has been implemented in many combinations of network and overlaying internetwork technologies, such as <a href="/wiki/IPv4" title="IPv4">IPv4</a>, <a href="/wiki/Chaosnet" title="Chaosnet">Chaosnet</a>, <a href="/wiki/DECnet" title="DECnet">DECnet</a> and Xerox <a href="/wiki/PARC_Universal_Packet" title="PARC Universal Packet">PARC Universal Packet</a> (PUP) using <a href="/wiki/IEEE_802" title="IEEE 802">IEEE 802</a> standards, <a href="/wiki/FDDI" title="FDDI" class="mw-redirect">FDDI</a>, <a href="/wiki/X.25" title="X.25">X.25</a>, <a href="/wiki/Frame_Relay" title="Frame Relay">Frame Relay</a> and <a href="/wiki/Asynchronous_Transfer_Mode" title="Asynchronous Transfer Mode">Asynchronous Transfer Mode</a> (ATM), IPv4 over <a href="/wiki/IEEE_802.3" title="IEEE 802.3">IEEE 802.3</a> and <a href="/wiki/IEEE_802.11" title="IEEE 802.11">IEEE 802.11</a> being the most common cases.</p> 
     <p>In <a href="/wiki/IPv6" title="IPv6">Internet Protocol Version 6</a> (IPv6) networks, the functionality of ARP is provided by the <a href="/wiki/Neighbor_Discovery_Protocol" title="Neighbor Discovery Protocol">Neighbor Discovery Protocol</a> (NDP).</p>

     <h2><span class="editsection">[<a href="/w/index.php?title=Address_Resolution_Protocol&amp;action=edit&amp;section=1" title="Edit section: Operating scope">edit</a>]</span> <span class="mw-headline" id="Operating_scope">Operating scope</span></h2> 
     <p>The Address Resolution Protocol is a request and reply protocol that runs encapsulated by the line protocol. It is communicated within the boundaries of a single network, never routed across internetwork nodes. This property places ARP into the <a href="/wiki/Link_Layer" title="Link Layer" class="mw-redirect">Link Layer</a> of the <a href="/wiki/Internet_Protocol_Suite" title="Internet Protocol Suite" class="mw-redirect">Internet Protocol Suite</a><sup id="cite_ref-2" class="reference"><a href="#cite_note-2"><span>[</span>2<span>]</span></a></sup>, while in the <a href="/wiki/Open_Systems_Interconnection" title="Open Systems Interconnection">Open Systems Interconnection</a> (OSI) model, it is often described as residing between Layers 2 and 3, being encapsulated by Layer 2 protocols. However, ARP was not developed in the OSI framework.</p> 
     <h2><span class="editsection">[<a href="/w/index.php?title=Address_Resolution_Protocol&amp;action=edit&amp;section=2" title="Edit section: Packet structure">edit</a>]</span> <span class="mw-headline" id="Packet_structure">Packet structure</span></h2> 
     <p>The Address Resolution Protocol uses a simple message format that contains one address resolution request or response. The size of the ARP message depends on the upper layer and lower layer address sizes, which are given by the type of networking protocol (usually <a href="/wiki/IPv4" title="IPv4">IPv4</a>) in use and the type of hardware or virtual link layer that the upper layer protocol is running on. The message header specifies these types, as well as the size of addresses of each. The message header is completed with the operation code for request (1) and reply (2). The payload of the packet consists of four addresses, the hardware and protocol address of the sender and receiver hosts.</p> 
     <p>The principal packet structure of ARP packets is shown in the following table which illustrates the case of IPv4 networks running on Ethernet. In this scenario, the packet has 48-bit fields for the sender hardware address (SHA) and target hardware address (THA), and 32-bit fields for the corresponding sender and target protocol addresses (SPA and TPA). Thus, the ARP packet size in this case is 28 bytes. The EtherType for ARP is 0x0806.</p>

     <h2><span class="editsection">[<a href="/w/index.php?title=Address_Resolution_Protocol&amp;action=edit&amp;section=3" title="Edit section: Example">edit</a>]</span> <span class="mw-headline" id="Example">Example</span></h2> 
     <p>For example, the computers <i>Matterhorn</i> and <i>Washington</i> are in an office, connected to each other on the office <a href="/wiki/Local_area_network" title="Local area network">local area network</a> by <a href="/wiki/Ethernet" title="Ethernet">Ethernet</a> cables and <a href="/wiki/Network_switches" title="Network switches" class="mw-redirect">network switches</a>, with no intervening <a href="/wiki/Gateway_(telecommunications)" title="Gateway (telecommunications)">gateways</a> or <a href="/wiki/Router_(computing)" title="Router (computing)">routers</a>. Matterhorn wants to send a packet to Washington. Through other means, it determines that Washington's IP address is 192.168.0.55. In order to send the message, it also needs to know Washington's MAC address. First, Matterhorn uses a cached ARP table to look up 192.168.0.55 for any existing records of Washington's MAC address (00:eb:24:b2:05:ac). If the MAC address is found, it sends the IP packet on the link layer to address 00:eb:24:b2:05:ac via the local network cabling. If the cache did not produce a result for 192.168.0.55, Matterhorn has to send a broadcast ARP message (destination FF:FF:FF:FF:FF:FF) requesting an answer for 192.168.0.55. Washington responds with its MAC address (00:eb:24:b2:05:ac). Washington may insert an entry for Matterhorn into its own ARP table for future use. The response information is cached in Matterhorn's ARP table and the message can now be sent.<sup id="cite_ref-7" class="reference"><a href="#cite_note-7"><span>[</span>7<span>]</span></a></sup></p> 
     <h2><span class="editsection">[<a href="/w/index.php?title=Address_Resolution_Protocol&amp;action=edit&amp;section=4" title="Edit section: ARP probe">edit</a>]</span> <span class="mw-headline" id="ARP_probe">ARP probe</span></h2> 
     <p>An <b>ARP probe</b> is an ARP request constructed with an all-zero <i>sender IP address</i>. The term is used in the <i>IPv4 Address Conflict Detection</i> specification (<a class="external mw-magiclink-rfc" href="//tools.ietf.org/html/rfc5227">RFC 5227</a>). Before beginning to use an IPv4 address (whether received from manual configuration, DHCP, or some other means), a host implementing this specification must test to see if the address is already in use, by broadcasting ARP probe packets.<sup id="cite_ref-8" class="reference"><a href="#cite_note-8"><span>[</span>8<span>]</span></a></sup></p> 
     <h2><span class="editsection">[<a href="/w/index.php?title=Address_Resolution_Protocol&amp;action=edit&amp;section=5" title="Edit section: ARP announcements">edit</a>]</span> <span class="mw-headline" id="ARP_announcements">ARP announcements</span></h2> 
     <p>ARP may also be used as a simple announcement protocol. This is useful for updating other hosts' mapping of a hardware address when the sender's IP address or MAC address has changed. Such an announcement, also called a <i>gratuitous ARP</i> message, is usually broadcast as an ARP request containing the sender's protocol address (SPA) in the target field (TPA=SPA), with the target hardware address (THA) set to zero. An alternative is to broadcast an ARP reply with the sender's hardware and protocol addresses (SHA and SPA) duplicated in the target fields (TPA=SPA, THA=SHA).</p> 
     <p>An ARP announcement is not intended to solicit a reply; instead it updates any cached entries in the ARP tables of other hosts that receive the packet. The operation code may indicate a request or a reply because the ARP standard specifies that the opcode is only processed after the ARP table has been updated from the address fields.<sup id="cite_ref-9" class="reference"><a href="#cite_note-9"><span>[</span>9<span>]</span></a></sup><sup id="cite_ref-10" class="reference"><a href="#cite_note-10"><span>[</span>10<span>]</span></a></sup><sup id="cite_ref-11" class="reference"><a href="#cite_note-11"><span>[</span>11<span>]</span></a></sup></p> 
     <p>Many operating systems perform gratuitous ARP during startup. That helps to resolve problems which would otherwise occur if, for example, a <a href="/wiki/Network_interface_controller" title="Network interface controller">network card</a> was recently changed (changing the IP-address-to-MAC-address mapping) and other hosts still have the old mapping in their ARP caches.</p> 
     <p>Gratuitous ARP is also used by some interface drivers to provide load balancing for incoming traffic. In a team of network cards, it is used to announce a different MAC address within the team that should receive incoming packets.</p> 
     <p>ARP announcements can be used to defend <a href="/wiki/Link-local_address" title="Link-local address">link-local IP addresses</a> in the <a href="/wiki/Zeroconf" title="Zeroconf" class="mw-redirect">Zeroconf</a> protocol (<a class="external mw-magiclink-rfc" href="//tools.ietf.org/html/rfc3927">RFC 3927</a>), and for IP address takeover within <a href="/wiki/High-availability_cluster" title="High-availability cluster">high-availability clusters</a>.</p> 
     <h2><span class="editsection">[<a href="/w/index.php?title=Address_Resolution_Protocol&amp;action=edit&amp;section=6" title="Edit section: ARP mediation">edit</a>]</span> <span class="mw-headline" id="ARP_mediation">ARP mediation</span></h2> 
     <p><b>ARP mediation</b> refers to the process of resolving Layer 2 addresses through a <a href="/wiki/Virtual_private_network#Virtual_private_wire_and_private_line_services_.28VPWS_and_VPLS.29" title="Virtual private network">Virtual Private Wire Service (VPWS)</a> when different resolution protocols are used on the connected circuits, e.g., <a href="/wiki/Ethernet" title="Ethernet">Ethernet</a> on one end and <a href="/wiki/Frame_Relay" title="Frame Relay">Frame Relay</a> on the other. In <a href="/wiki/IPv4" title="IPv4">IPv4</a>, each <a href="/wiki/Provider_Edge" title="Provider Edge">Provider Edge</a> (PE) device discovers the IP address of the locally attached <a href="/wiki/Customer_edge" title="Customer edge">Customer Edge</a> (CE) device and distributes that IP address to the corresponding remote PE device. Then each PE device responds to local ARP requests using the IP address of the remote CE device and the hardware address of the local PE device. In <a href="/wiki/IPv6" title="IPv6">IPv6</a>, each PE device discovers the IP address of both local and remote CE devices and then intercepts local <a href="/wiki/Neighbor_Discovery_Protocol" title="Neighbor Discovery Protocol">Neighbor Discovery</a> (ND) and Inverse Neighbor Discovery (IND) packets and forwards them to the remote PE device.<sup id="cite_ref-12" class="reference"><a href="#cite_note-12"><span>[</span>12<span>]</span></a></sup></p> 
     <h2><span class="editsection">[<a href="/w/index.php?title=Address_Resolution_Protocol&amp;action=edit&amp;section=7" title="Edit section: Inverse ARP and Reverse ARP">edit</a>]</span> <span class="mw-headline" id="Inverse_ARP_and_Reverse_ARP">Inverse ARP and Reverse ARP</span></h2> 
     <p><b>Inverse Address Resolution Protocol</b> (Inverse ARP or InARP) is used to obtain <a href="/wiki/Network_Layer" title="Network Layer" class="mw-redirect">Network Layer</a> addresses (for example, <a href="/wiki/IP_address" title="IP address">IP addresses</a>) of other nodes from <a href="/wiki/Data_Link_Layer" title="Data Link Layer" class="mw-redirect">Data Link Layer</a> (Layer 2) addresses. It is primarily used in <a href="/wiki/Frame_Relay" title="Frame Relay">Frame Relay</a> (<a href="/wiki/DLCI" title="DLCI" class="mw-redirect">DLCI</a>) and ATM networks, in which Layer 2 addresses of <a href="/wiki/Virtual_circuit" title="Virtual circuit">virtual circuits</a> are sometimes obtained from Layer 2 signaling, and the corresponding Layer 3 addresses must be available before those virtual circuits can be used.<sup id="cite_ref-13" class="reference"><a href="#cite_note-13"><span>[</span>13<span>]</span></a></sup></p> 
     <p>Since ARP translates Layer 3 addresses to Layer 2 addresses, InARP may be described as its inverse. In addition, InARP is implemented as a protocol extension to ARP: it uses the same packet format as ARP, but different operation codes.</p> 
     <p>The <a href="/wiki/Reverse_Address_Resolution_Protocol" title="Reverse Address Resolution Protocol">Reverse Address Resolution Protocol</a> (Reverse ARP or RARP), like InARP, translates Layer 2 addresses to Layer 3 addresses. However, in InARP the requesting station queries the Layer 3 address of another node, whereas RARP is used to obtain the Layer 3 address of the requesting station itself for address configuration purposes. RARP is obsolete; it was replaced by <a href="/wiki/BOOTP" title="BOOTP" class="mw-redirect">BOOTP</a>, which was later superseded by the <a href="/wiki/Dynamic_Host_Configuration_Protocol" title="Dynamic Host Configuration Protocol">Dynamic Host Configuration Protocol</a> (DHCP).<sup id="cite_ref-14" class="reference"><a href="#cite_note-14"><span>[</span>14<span>]</span></a></sup></p> 
     <h2><span class="editsection">[<a href="/w/index.php?title=Address_Resolution_Protocol&amp;action=edit&amp;section=8" title="Edit section: ARP spoofing and Proxy ARP">edit</a>]</span> <span class="mw-headline" id="ARP_spoofing_and_Proxy_ARP">ARP spoofing and Proxy ARP</span></h2>

      <div class="thumbinner" style="width:202px;">

     <p>Because ARP does not provide methods for authenticating ARP replies on a network, ARP replies can come from systems other than the one with the required Layer 2 address. An ARP <i>proxy</i> is a system which answers the ARP request on behalf of another system for which it will forward traffic, normally as a part of the network's design, such as for a dialup internet service. By contrast, in ARP <i>spoofing</i> the answering system, or <i>spoofer</i>, replies to a request for another system's address with the aim of intercepting data bound for that system. A malicious user may use ARP spoofing to perform a <a href="/wiki/Man-in-the-middle" title="Man-in-the-middle" class="mw-redirect">man-in-the-middle</a> or <a href="/wiki/Denial-of-service" title="Denial-of-service" class="mw-redirect">denial-of-service</a> attack on other users on the network. Various software exists to both detect and perform ARP spoofing attacks, though ARP itself does not provide any methods of protection from such attacks.<sup id="cite_ref-grc_15-0" class="reference"><a href="#cite_note-grc-15"><span>[</span>15<span>]</span></a></sup></p> 
     <h2><span class="editsection">[<a href="/w/index.php?title=Address_Resolution_Protocol&amp;action=edit&amp;section=9" title="Edit section: Alternatives to ARP">edit</a>]</span> <span class="mw-headline" id="Alternatives_to_ARP">Alternatives to ARP</span></h2> 
     <p>Each computer maintains its own table of the mapping from <a href="/wiki/Layer_3" title="Layer 3" class="mw-redirect">Layer 3</a> addresses (e.g. <a href="/wiki/IP_addresses" title="IP addresses" class="mw-redirect">IP addresses</a>) to <a href="/wiki/Layer_2" title="Layer 2" class="mw-redirect">Layer 2</a> addresses (e.g. <a href="/wiki/Ethernet" title="Ethernet">ethernet</a> <a href="/wiki/MAC_addresses" title="MAC addresses" class="mw-redirect">MAC addresses</a>). In a modern computer this is maintained almost entirely by ARP packets on the local network and it thus often called the 'ARP cache' as opposed to 'Layer 2 address table'. In older computers, where broadcast packets were considered an expensive resource, other methods were used to maintain this table, such as static configuration files,<sup id="cite_ref-16" class="reference"><a href="#cite_note-16"><span>[</span>16<span>]</span></a></sup> or centrally maintained lists. Since at least the 1980s<sup id="cite_ref-17" class="reference"><a href="#cite_note-17"><span>[</span>17<span>]</span></a></sup> networked computers have had a command called <i>arp</i> for interrogating or manipulating this table, and practically all modern personal computers have a variant of this.<sup id="cite_ref-18" class="reference"><a href="#cite_note-18"><span>[</span>18<span>]</span></a></sup><sup id="cite_ref-19" class="reference"><a href="#cite_note-19"><span>[</span>19<span>]</span></a></sup><sup id="cite_ref-20" class="reference"><a href="#cite_note-20"><span>[</span>20<span>]</span></a></sup><sup id="cite_ref-21" class="reference"><a href="#cite_note-21"><span>[</span>21<span>]</span></a></sup></p> 
     <h2><span class="editsection">[<a href="/w/index.php?title=Address_Resolution_Protocol&amp;action=edit&amp;section=10" title="Edit section: ARP Stuffing">edit</a>]</span> <span class="mw-headline" id="ARP_Stuffing">ARP Stuffing</span></h2> 
     <p>Embedded systems such as networked cameras<sup id="cite_ref-22" class="reference"><a href="#cite_note-22"><span>[</span>22<span>]</span></a></sup> and networked power distribution devices,<sup id="cite_ref-23" class="reference"><a href="#cite_note-23"><span>[</span>23<span>]</span></a></sup> which lack a user interface, can use so-called <i>ARP stuffing</i> to make an initial network connection, although this is a misnomer as there is no ARP <i>protocol</i> involved. This is a solution to an issue in network management of consumer devices, specifically the allocation of IP addresses of ethernet devices where 1) the user doesn't have the ability to control <a href="/wiki/DHCP" title="DHCP" class="mw-redirect">DHCP</a> or similar address allocation protocols, 2) the device doesn't have a user interface to configure it, and 3) the user's computer can't communicate with it because it has no suitable IP address.</p> 
     <p>The solution adopted is as follows: the user's computer has an IP address <i>stuffed</i> manually into its address table (normally with the <i>arp</i> command with the MAC address taken from a label on the device) and then sends special packets to the device, typically a <a href="/wiki/Ping_(networking_utility)" title="Ping (networking utility)">ping</a> packet with a non-default size. The device then adopts this IP address, and the user then communicates with it by <a href="/wiki/Telnet" title="Telnet">telnet</a> or <a href="/wiki/Http" title="Http" class="mw-redirect">web</a> protocols to complete the configuration. Such devices typically have a method to disable this process once the device is operating normally, as it is open to <a href="/wiki/Denial_of_Service" title="Denial of Service" class="mw-redirect">Denial of Service</a> attack.</p>
